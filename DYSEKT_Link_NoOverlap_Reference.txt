This document covers two inter-related slice editing features in DYSEKT: the LINK toggle (which locks START and END knobs so they move together, preserving slice length) and the No-Overlap constraint (which prevents adjacent slice boundaries from crossing each other). It explains what each feature does, why it exists, how they differ, when to use each, and the exact C++ implementation strategy within the DYSEKT codebase.

1. Overview


1.1 The Problem Space
A DYSEKT slice is defined by two sample-position integers: startSample and endSample. The user edits these via the START and END knobs in the Slice Control Bar, and by dragging slice edges directly on the waveform. Two distinct usability problems emerge from this simple model:

Problem 1: Moving a slice without resizing it
If a user wants to shift a slice earlier or later in the timeline, they must move START and END by exactly the same amount. With two independent knobs this is nearly impossible to do accurately by hand - even a 1-sample difference changes the slice length.

Problem 2: Preventing slices from crossing each other
When dragging an edge, nothing stops it from moving past a neighbouring edge, which would invert the slice (endSample < startSample) or cause two slices to overlap - both invalid states that crash the audio engine or produce silence.

1.2 The Two Features
DYSEKT addresses these problems with two separate mechanisms that operate at different layers of the stack and serve different purposes:

2. The LINK Feature


2.1 What It Does
When LINK is active, any change to START also applies the same delta to END, and vice versa. The slice behaves as a rigid window - it translates along the waveform without changing length. LINK is a toggle: one click enables it, another disables it. Its state is stored in an atomic boolean on the processor (processor.slicesLinked) so it is accessible safely from both the UI and audio threads.

2.2 User Workflow

- Repositioning a slice
Enable LINK, then scrub the START knob. The entire slice moves with it. Disable LINK when the new position is found.

- Auditioning different regions
Lock slice length, enable LINK, and use the MIDI NOTE to trigger the slice while scrubbing START - effectively scrubbing through the sample in fixed windows, like a looping playhead.

- Batch shifting
If multiple slices share a layout that needs to be shifted earlier or later (e.g. after a sample was trimmed at the start), LINK on each one in turn avoids accidental resize.

2.6 Edge Cases & Clamping



DYSEKT / Link & No-Overlap: Technical Reference


- Slice at the start of the sample (startSample == 0)
If LINK is active and the user drags START left past 0, the clamp at 0 will shorten the delta applied to END. Both boundaries stop moving when START hits 0. This is correct - the slice "jams" against the left wall.

- Slice at the end of the sample
Symmetrically, when END hits sampleNumFrames the slice stops translating rightward. START is also clamped to maxLen - sliceLength so the length is still preserved as long as there is room.

- LINK toggled mid-drag
LINK state is read once per command execution on the audio thread. If the user clicks LINK during a knob drag, the next command will use the new state. There is no partial-apply problem because each command is fully self-contained.

- Multiple slices
LINK applies only to the currently selected slice. It does not gang-move all slices simultaneously - that would require a separate "global shift" command not yet implemented.

3. The No-Overlap Constraint


3.1 What It Does
No-Overlap is a drag-time guard that prevents a slice edge from being dragged past its neighbour. When dragging the left edge of slice N, it cannot move right of the left edge of slice N's end, and cannot move left past the right edge of slice N-1. Similarly for the right edge. The constraint is enforced silently - the edge simply stops moving at the boundary rather than crossing it.

3.2 Why It Existed
Early in DYSEKT's development, the SliceManager had no validation against inverted or overlapping slices. An inverted slice (startSample > endSample) would cause the audio engine to read negative sample counts, producing undefined behaviour or silence. No-Overlap was a defensive UI guard to make that state unreachable through normal interaction.

3.4 Why It Was Removed
The constraint created two friction points that outweighed its safety benefit once the SliceManager was made robust:

- Frustrating for intentional overlap workflows
Certain techniques - crossfade editing, overlapping loops, granular layering - require adjacent slices to share sample ranges. The clamp made these impossible without a workaround.

- Confusing drag behaviour near boundaries
When the cursor crossed the neighbour boundary, the edge would suddenly stop while the cursor kept moving - creating a "sticky" feel that confused users into thinking the drag had ended.

- SliceManager now validates at apply time
clampSlicesToSampleBounds() and rebuildMidiMap() in the processor already prevent truly invalid states (start >= end, out-of-buffer positions) at the command-apply layer. The UI guard was redundant.

4.2 Remaining Work - LINK During Waveform Drag
LINK is correctly enforced when the user edits START/END via the knobs in SliceControlBar. However, when the user drags a slice edge directly on the waveform in WaveformView, LINK is not yet applied. The drag updates dragPreviewStart and dragPreviewEnd independently, then pushes a CmdSetSliceBounds which the processor handles - but the preview positions shown during the drag do not reflect the linked constraint, so the display is misleading even if the final result is correct.
